Hoot Mobile Changelog
=====================

Version 0.2.0
-------------

Date: 2026-06-19

Overview:

  * Advanced the app from the previous GitHub state at commit 04a0bdf
    ("Removed non-functional navigation buttons so the app works without
    crashing") to a modernized Expo SDK 56 / React Native 0.85 app.

  * Updated app.json from version 0.1.2 to 0.2.0 and added the Expo plugin
    declarations required by the current native modules.

  * Added package-lock.json so the upgraded npm dependency graph is recorded
    reproducibly.

  * Added file headers, section banners, and end-of-file markers across the
    TypeScript and TSX source files touched by this pass.

  * Added the Hitide/Lotide 0.18.0 compatibility pass for source feeds,
    private messages, and user follows while keeping the new surfaces gated for
    older servers.

Lotide API compatibility:

  * Matched community browsing to the current Lotide server behavior:
    list requests now use the community scopes, accepted-follow filtering,
    authenticated include_your data, alphabetic sorting, and server pagination
    forms used by the real handler.

  * Updated community follow handling to match Lotide's response body.
    Follow requests now wait briefly for remote acceptance and validate the
    returned accepted flag instead of assuming success from an empty response.

  * Changed community editing to send description_markdown, which is the field
    accepted by the Lotide community PATCH endpoint.

  * Fixed community description editing when the server returns a structured
    content object. The edit screen now starts from markdown, then text, then
    HTML content, instead of passing the content object directly to the text
    input.

  * Adjusted follow feedback so a false accepted value is treated as "not
    accepted yet" rather than a hard rejection.

  * Updated post listing requests to include current aggregate filter behavior,
    optional authenticated include_your data, encoded page tokens, community
    filters, and sticky sorting for community hot feeds.

  * Updated post and comment reply requests to use the current include_your,
    sort, pagination, and response-normalization behavior.

  * Updated post, comment, community, and reply creation to validate returned
    ids before the app stores or navigates to the new resource.

  * Added Lotide user activity support through the users/:id/things endpoint.

  * Added moderation support for moderated-community lists, pending community
    flags, and community flag dismissal.

  * Added gated Lotide 0.18 collection target support through the
    collection_targets API. Hoot can now list source feeds, load source-feed
    details, read cached source items, follow and unfollow source feeds, like
    and unlike source preview items, and comment on source items when the
    backend reports a compatible reply path. Source lists now also expose the
    server's software-count filters so users can narrow feeds by the supported
    Lotide 0.18 source families.

  * Added gated Lotide 0.18 private-message support through
    users/~me/messages. Hoot can now load the conversation list, load threads,
    send replies chained to the latest known message, and dismiss conversations
    until new activity arrives.

  * Added gated Lotide 0.18 user-follow support through the new
    users/:id/follow and users/:id/unfollow endpoints. Other-user profile
    activity screens can now show the remote profile header, follow or unfollow
    the user, and open a private-message thread when the server supports the
    required APIs.

Response handling and validation:

  * Added services/LotideService/validation.ts to normalize untrusted Lotide
    payloads before UI code consumes them.

  * Made instance metadata loading tolerant of older or incomplete server
    software fields while preserving the parsed API version used elsewhere in
    the app.

  * Normalized current Lotide community fields, including your_follow,
    federation status, discovery metadata, latest unfollow status, visibility
    suppression, last post summaries, remote post counts, and pending
    moderation action counts.

  * Normalized post and comment fields supplied by the server, including
    remote_url, content_markdown, sensitive flags, comment attachments,
    comment deletion state, and federation status.

  * Kept typed Lotide notifications and user activity records aligned with the
    server's serialized response enums, including user follow notifications.

  * Added runtime normalization for Lotide collection target lists, source
    details, source item reader responses, source item comments, source item
    votes, and private-message records.

  * Added centralized feature gates for source feeds, private messages, and
    user follows, plus the existing user-follow notification gate, so partial
    or older Lotide instances degrade by hiding unsupported mobile entry points
    rather than failing at navigation time.

  * Validated Lotide 0.18 mutation responses that Hoot depends on before
    updating local UI state. Source-item comments must return a usable id,
    private-message sends must return a private-message record, and
    private-message dismissals must confirm dismissal.

  * Added a shared utility for converting unknown thrown values into user-safe
    alert messages.

  * Added a shared 30-second timeout to Lotide API requests. Stalled servers or
    broken network paths now fail through the normal error and retry surfaces
    instead of leaving screens in an indefinite loading state.

  * Hardened Lotide endpoint URL construction. API base URLs are now trimmed
    and joined to endpoint paths with exactly one slash, so a saved custom
    server ending in "/" cannot turn every request into a double-slash URL.

  * Hardened the shared refresh hook so caller cleanup functions run when a
    refresh, dependency change, or unmount replaces an in-flight load. Stale
    stop-loading callbacks from replaced loads can no longer clear the active
    loading state.

  * Hardened startup synchronization so transient instance/profile failures no
    longer erase a valid stored login. Explicit authentication failures now
    expire the login while preserving the selected server context.

  * Hardened root startup/session recovery so stored-context read failures are
    logged through diagnostics instead of becoming unhandled startup promises,
    and expired logins still clear the active persisted context even if saved
    account bookkeeping fails.

  * Added defensive AsyncStorage parsing for active context and saved account
    records. Corrupt JSON now self-heals to a safe signed-out state instead of
    crashing startup or the account picker. Active contexts and saved profiles
    now canonicalize Lotide API URLs so legacy trailing-slash profile aliases
    collapse back to one account row.

  * Added a root React render error boundary. If a screen crashes while
    rendering, Hoot now shows a touch-friendly recovery view with a retry action
    instead of leaving the app on a blank or permanently broken render tree.

  * Added centralized diagnostic logging through utils/debugLog.ts. Startup,
    render-boundary, Lotide request, and background notification warnings/errors
    now pass through one formatting and failure-isolation point instead of
    writing directly to console APIs throughout the app.

Notification reliability:

  * Added Android local notification support with expo-notifications.

  * Replaced the deprecated expo-background-fetch scheduler with the current
    expo-background-task API for local notification polling.

  * Added services/LotideNotificationPoller.ts to register the background task,
    manage the Android notification channel, request notification permission,
    store the per-account notification baseline, and surface new notifications
    as local alerts.

  * Changed phone notification dedupe to use Hoot's own per-account local
    notification baseline instead of relying only on Lotide's global unseen
    flag. This lets Hoot notify for a phone-new notification even if Hitide or
    another client already fetched the notifications endpoint first.

  * Made empty notification baselines explicit. A newly enabled account with no
    current notifications now still alerts for the next phone-new notification,
    even if another client has cleared Lotide's unseen flag by the time Hoot
    polls.

  * Added app startup and foreground-resume notification polling so enabled
    Android devices do not have to wait only for the operating system's
    background task window to notice new Lotide notifications.

  * Raised the Android Lotide notification channel to high importance and added
    default notification sound to local alerts.

  * Versioned the Android notification channel as lotide-notifications-v2 so
    upgraded installs are not stuck with an older default-importance channel
    that Android will not raise programmatically.

  * Moved notification permission prompting into the enablement transaction.
    Enabling notifications now creates the channel and asks the OS permission
    before creating the local baseline or storing the enabled flag.

  * Preserved the post and comment records embedded in Lotide notification
    responses so the notification screen can render freshly returned
    notifications before the normal post/comment caches have seen them.

  * Distinguished reply and mention notification text in both the notification
    screen and local Android alert titles.

  * Added notification poller tests covering enable-time baselining,
    denied OS notification permission, cross-client unseen clearing, and
    first-run old-history suppression.

  * Bounded the per-account notification dedupe cache and made it current-page
    first. Large notification histories now keep the freshly fetched Lotide
    notification fingerprints before older local cache entries, preventing
    background polling from immediately forgetting notifications it just saw.

  * Capped large notification batches so a background poll cannot flood the
    Android shade with one local alert per backlog item. Hoot now sends a small
    set of routed individual alerts plus a summary alert that opens the
    notifications tab for the rest.

  * Added an Android-only settings toggle for background notifications and
    updated the settings text to describe OS-controlled background checks
    rather than promising a fixed polling interval.

  * Added Android notification diagnostics to settings. The app now reports
    local-alert permission status, background-polling availability, background
    task registration, and exposes a test-notification action that schedules a
    local alert through the same high-importance Lotide channel.

  * Added blocked-permission recovery for Android notifications. Diagnostics now
    preserve whether the OS can still be asked for notification permission, and
    settings offers a direct notification-settings shortcut when Android reports
    that local alerts are blocked in system settings.

  * Added persisted notification-poll diagnostics. Hoot now records the last
    poll attempt, last successful poll, last local-alert count, skipped-poll
    reason, and last poll error so a phone can prove whether the notification
    path is fetching, scheduling, or failing.

  * Made Android background notification wakes observable when no signed-in
    account is available. A background task that wakes after logout or before
    context restoration now records a "signed out" skip reason instead of
    disappearing from notification diagnostics.

  * Added a settings action to check Lotide notifications immediately. This
    gives phone testing a direct server-poll path instead of depending only on
    startup, foreground-resume, or Android's background task window.

  * Added notification tap routing for both warm app and cold-start launches.
    Reply notifications now open the related post with the reply highlighted;
    user-follow notifications open the notification list.

  * Added private-message notification support. Lotide 0.18 private-message
    notifications now render in the notification screen, produce local Android
    alerts, and route notification taps directly to the sender conversation.

  * Expanded notification poller coverage for Expo's current cold-start
    notification response cache and defensive response clearing path.

  * Kept previously loaded notifications visible when a notification-list
    refresh fails. The screen now shows the reload failure inline above the
    existing notifications instead of replacing useful notification history
    with an empty retry state.

  * Added a foreground notification handler using the current Expo
    shouldShowBanner and shouldShowList fields.

  * Made notification enablement transactional. The app now verifies local
    notification permission and creates the local notification baseline before
    storing the enabled flag, and rolls back the setting if registration fails.

  * Made task-registration availability part of the user-facing enablement
    transaction. If Android reports background tasks unavailable, the setting is
    left disabled instead of appearing enabled without a registered task.

  * Hardened notification diagnostics in settings so delayed permission or
    background-task checks are ignored after the user navigates away from the
    screen. This prevents late Android diagnostics from raising stale alerts or
    updating an unmounted settings view.

  * Hardened Android notification actions in settings. Manual notification
    checks, local test notifications, notification toggle changes, and the
    Android settings shortcut now have synchronous in-flight guards so fast
    repeated taps cannot schedule duplicate notification work before React
    disables the visible control.

  * Remaining limitation: this is still local polling. If the app is
    user-force-killed, Android and iOS may not run background tasks. True
    killed-app delivery still needs server push support; Lotide's current
    notification subscription endpoint is a compatibility stub.

Navigation, screens, and flows:

  * Updated React Navigation usage for the current v7 packages, including
    typed route params, typed navigation calls, and a navigation theme derived
    from the app color tokens.

  * Replaced the old placeholder linking map with Hoot routes for feeds,
    search, new posts, notifications, profile, posts, comments, communities,
    profile activity, moderation, settings, and not-found fallback paths.

  * Added gated Feeds navigation for Lotide 0.18 source feeds, including the
    source list tab/drawer entry, source detail screen, source item reader, and
    deep links for source detail and source item routes.

  * Added gated Messages navigation for Lotide 0.18 private messages, including
    the messages tab/drawer entry, conversation thread screen, and deep links
    for conversation routes.

  * Added ProfileActivityScreen.tsx to display the user's mixed post and
    comment activity from Lotide, including pagination and navigation back to
    related posts.

  * Added pull-to-refresh behavior to private-message inboxes and conversation
    threads. Failed inbox refreshes now keep the previously visible
    conversations on screen while surfacing the reload failure.

  * Added paged private-message inbox loading. Conversation lists now request
    later Lotide pages as the user scrolls, suppress duplicate in-flight page
    loads, and deduplicate overlapping conversation previews by conversation
    partner before rows are appended.

  * Added paged private-message thread loading. Longer conversations now expose
    a touch-friendly "Load earlier messages" action, suppress duplicate
    in-flight page loads, and deduplicate overlapping message pages by message
    id.

  * Hardened private-message sending so a server-accepted message is inserted
    into the visible conversation immediately, even if the follow-up thread
    reload fails. Failed sends now keep the draft and show the server error
    message when one is available.

  * Hardened private-message send submission with a synchronous pending guard.
    Very fast repeated taps on the Send button can no longer submit duplicate
    messages before React rerenders the disabled state.

  * Hardened private-message thread cleanup so delayed page loads and message
    sends are ignored after the user leaves the conversation. Failed sends can
    no longer raise stale alerts from a thread screen that is no longer active.

  * Hardened private-message conversation dismissal. Dismiss buttons now disable
    while the request is pending, duplicate taps cannot submit duplicate
    dismissals, accepted dismissals remove the conversation immediately, stale
    follow-up inbox reloads cannot resurrect the dismissed preview, newer
    activity from the same sender still reopens the conversation, and delayed
    failures after leaving the inbox no longer show stale alerts.

  * Added honest pull-to-refresh behavior to the source-feed list. Failed source
    list refreshes now keep the previously visible feeds on screen and show the
    reload failure inline.

  * Hardened source-feed follow and unfollow actions from both the feed list
    and source detail screen. Pending follow changes now disable the active
    action button, duplicate taps cannot create duplicate requests, and late
    failures after leaving the source view no longer show stale alerts or
    trigger dismissed refreshes.

  * Added pull-to-refresh behavior to Lotide 0.18 source-feed detail and source
    item reader screens. Failed detail/item refreshes now keep the previously
    visible feed or cached item on screen while showing the reload failure
    inline.

  * Hardened Lotide 0.18 source-item comment submission so an accepted comment
    appears in the cached item immediately, even if the follow-up item reload
    fails. Source-item comments now also prevent duplicate pending submits,
    ignore delayed submit results after the item reader is dismissed, and keep
    failed drafts visible with the server error message when one is available.

  * Hardened post/comment voting and Lotide 0.18 source-item likes. Regular
    post and comment votes now update optimistically, block duplicate pending
    vote requests, roll back failed mutations, and avoid stale alerts after a
    voting surface unmounts. Source-feed preview and source-item likes now
    disable the pending heart action, suppress duplicate like requests, show
    server error details, and ignore late failures or refreshes after the user
    leaves the source view.

  * Hardened the regular post/comment reply composer. Replies now submit
    trimmed text, prevent blank or duplicate submissions, keep drafts on
    failure, show server error details, ignore delayed submit results after the
    composer is dismissed, and return to the post with the accepted comment
    highlighted.

  * Added pull-to-refresh behavior to profile activity timelines. Failed
    activity refreshes now keep the previously visible activity and profile
    header on screen while surfacing the reload failure inline.

  * Hardened profile activity pagination. Repeated FlatList end-reaching while
    a page is already in flight no longer issues duplicate Lotide requests, and
    overlapping activity pages are deduplicated by activity type and id before
    rows are appended.

  * Hardened profile follow and unfollow actions on other-user activity
    screens. Pending user-follow changes now disable the active button,
    duplicate taps cannot create duplicate requests, and late failures after
    leaving the profile no longer show stale alerts or trigger dismissed
    refreshes.

  * Added ModerationScreen.tsx to show moderated communities, pending flags,
    flagged posts, flagger information, and flag dismissal.

  * Hardened moderation flag dismissal. Pending dismissals now disable the
    active action, duplicate taps cannot submit duplicate flag updates, failed
    dismissals show the server error reason when one is available, delayed
    dismiss results after leaving the moderation dashboard are ignored, and
    stale moderation community or flag loads can no longer update a dismissed
    screen.

  * Reworked ProfileScreen to load profile data defensively, render structured
    profile descriptions, link to activity/moderation/settings, and handle
    logout when the stored login user record is missing.

  * Added pull-to-refresh behavior to the signed-in profile screen. Failed
    profile or followed-community refreshes now keep the previously visible
    profile data on screen and show the reload failure inline.

  * Hardened profile account switching and logout so the active persisted
    Lotide context is removed before the UI reports that the account is no
    longer active. Background notification polling can no longer keep using a
    locally logged-out account after restart or Android background wake-up.

  * Hardened signed-in profile account actions against repeated taps and stale
    failures. Switch-account and logout rows now disable while storage or
    logout work is pending, duplicate logout prompts are ignored, and delayed
    account-action failures after leaving the profile no longer raise stale
    alerts.

  * Reworked SearchScreen and CommunityFinder so community browsing has Mine
    and Everything tabs backed by the current Lotide community list API.

  * Hardened the shared CommunityFinder picker used by community search and
    new-post community selection. Slow loads now show an explicit loading
    state, failed loads show the shared retry control instead of a blank list,
    and empty filtered results explain that no matching communities were found.

  * Added all-page community loading with repeated-cursor detection and a page
    safety limit. Community search, new-post community selection, and the
    profile followed-community list now see communities beyond the first page.

  * Restored dev.narwhal.city to the seeded login host list after checking the
    old GitHub host list against live Lotide instance endpoints.

  * Hardened the first-run host and credential flow. Custom host entry now has
    a visible Continue action, pasted host URLs are normalized to the host
    domain before selection, empty host submissions are rejected, login and
    registration trim user-facing identifiers, and in-flight credential
    submissions cannot be duplicated.

  * Hardened saved-profile activation and seeded-host probing on the first-run
    host picker. Stored account rows now disable while an activation is
    pending, duplicate activation taps are ignored, delayed activation failures
    after leaving the picker no longer raise stale alerts, saved-profile loads
    ignore unmounted results, and known-host probe results are applied by
    domain instead of by the currently visible filtered index.

  * Hardened login and registration cleanup so delayed authentication
    completions are ignored after the user leaves the credential form. A late

    can no longer raise an alert from a dismissed form.

  * Made account activation durable immediately after successful login,
    registration, or saved-profile selection. The active Lotide context is now
    written to local storage before Redux reports the account active, so app
    restart and Android background notification polling see the same account as
    the visible UI.

  * Reworked CommunityScreen so it can load a community by id from deep links,
    render structured descriptions, refresh community metadata, and show retry
    states for community or feed failures.

  * Hardened CommunityScreen follow and unfollow actions. Signed-out readers
    no longer see community action buttons, pending follow changes disable the
    action button, duplicate taps cannot create duplicate requests, and late
    failures after leaving the screen no longer show stale alerts or reload the
    dismissed community view.

  * Reworked FeedScreen, ModalScreen, PostDisplay, and CommentsDisplay to show
    explicit retry/loading states instead of failing silently or crashing when
    posts, comments, authors, or communities are missing.

  * Reworked the NotFound fallback screen to use the app theme and shared
    touch-friendly button styling instead of a hard-coded white route page.

  * Made feed loading key-aware so sort, community, follow-scope, and account
    changes reset visible feed state and stale in-flight requests cannot append
    into a newer feed or dispatch after the feed hook unmounts.

  * Made comment pagination deduplicate overlapping server pages so post
    comments and nested replies do not render duplicate rows when a backend
    repeats an id across page boundaries.

  * Made comment loading ignore stale in-flight responses and failures so a
    slow older page load cannot overwrite a newer refresh or nested-reply
    request, and cannot write into Redux after the comment hook unmounts.

  * Made single post and single comment loading ignore stale responses so older
    detail loads cannot overwrite a newer reload or write into Redux after the

  * Added persisted app settings for the default feed sort. Settings now

    servers until the connected API supports Top sorting.

  * Split explicit "not in your follows" feed requests from omitted follow
    filters in the feed-state key, and prevented the signed-out feed prompt from
    starting an invisible authenticated feed request.

  * Updated post sharing to use the post remote_url when available and otherwise
    derive the post URL from the current Lotide instance instead of hard-coding
    a development host.

  * Reworked NotificationScreen to handle reply notifications, user-follow
    notifications, private-message notifications, empty states, fetch failures,
    and navigation to highlighted replies, follower profiles, or conversation
    threads.

  * Reworked ProfileActivityScreen for other users so it can load profile
    header data, show follow state, follow or unfollow users on API 18 servers,

    available.

  * Reworked SettingsScreen to remove the deprecated react-native-settings-screen
    dependency and provide an in-app settings UI for the Lotide API URL,
    persisted feed-sort preference, and Android notification toggle.

  * Hardened SettingsScreen saves. API URLs are trimmed and canonicalized before
    persistence, malformed values are rejected with the same http:// or https://
    contract shown to users, duplicate save taps are ignored while storage is
    pending, and the API URL input now exposes an accessibility label.

  * Updated NewPostScreen, NewCommunity, EditCommunityScreen, and
    ForgotPasswordScreen for current React Native props, better error messages,
    and safer navigation after successful requests.

  * Hardened NewPostScreen submission so duplicate taps cannot create duplicate
    post requests, post fields are trimmed before submission, and delayed
    created-post reloads are ignored after the composer is dismissed.

  * Hardened community creation and editing forms. New community names and
    descriptions are trimmed before API calls, signed-out users are sent to the
    login prompt, in-flight create/edit submits are disabled, and a created
    community still opens when optional description save, follow, or reload work
    fails after the server has already accepted the community. Delayed
    create/edit successes and failures are now ignored after the user leaves
    the form, so stale requests cannot navigate, alert, or update a dismissed
    screen.

  * Hardened ForgotPasswordScreen as a real mobile reset flow. Email and reset
    keys are trimmed before API calls, duplicate key/reset submits are blocked,
    key rejection is shown inline, reset inputs are themed and touch-friendly,
    and the old developer-only "this shouldn't happen" alert was replaced with
    recoverable user-facing text.

  * Hardened password-reset cleanup so delayed key requests, key checks, and
    reset submissions are ignored after the reset screen is dismissed. Late
    failures can no longer raise stale alerts, and a late reset success can no
    longer navigate from a screen the user has already left.

  * Added user-visible failure handling for vote requests, comment submission,
    post-create reloads, password reset key checks, and password reset submits.

Rendering, theme, and UI behavior:

  * Replaced react-native-htmlview with react-native-render-html. The old
    package was already at its latest version but still pulled deprecated React
    Native prop-type shims; the new renderer uses the current source,
    renderersProps, ignoredDomTags, and contentWidth APIs.

  * Reworked ContentDisplay to choose HTML, markdown-derived text, or plain
    text safely, truncate previews before rendering, support current custom
    renderers for abbr/details/img, ignore unsafe tags, and open links through
    React Native Linking.

  * Stabilized ContentDisplay's react-native-render-html configuration objects
    across rerenders so post, comment, and message bodies avoid needless render
    engine rebuilds while preserving typed renderer styles.

  * Hardened HrefDisplay link previews. External preview links now normalize
    obvious bare web URLs, direct image previews open from the image itself,
    unsupported schemes fall back visibly instead of being treated as images,
    failed platform browser handoffs show the original link, and malformed
    image dimensions are clamped before they reach layout styles.

  * Centralized external-link opening through utils/externalLink.ts so HTML
    content, post previews, source feeds, and source items all share the same
    guarded URL normalization, unsupported-scheme fallback, and platform-open
    failure handling.

  * Hardened the shared haptic feedback wrapper. Hoot now skips haptics on web,
    delegates native feedback through one service, and treats native haptic
    failures as diagnostic-only so a failed vibration cannot become an
    interaction error.

  * Added RetryState.tsx as the shared accessible retry component used by
    feeds, comments, posts, notifications, profiles, moderation, host loading,
    and community loading.

  * Reworked the color system into explicit typed light/dark palettes and
    switched the dark theme to true black with app-controlled tab-bar colors.

  * Updated components for safer optional data handling, including unknown
    authors, unknown communities, missing login users, disabled navigation
    targets, and API load failures.

  * Added accessibility labels, roles, selected/disabled states, and clearer
    press handling to tabs, host/profile selectors, post/community links, and
    retry actions.

  * Tightened feed and login accessibility semantics. Feed rows now expose a
    labeled open-post button action, comment rows describe their action-toggle
    behavior, and static login host headers no longer create fake button
    targets for assistive technologies.

  * Centralized the supported Lotide API version floor so startup checks and the
    known-host picker agree that API version 8 is the minimum supported server
    surface.

  * Updated SwipeAction, Hoverable, HoverHighlight, TappableList, VoteCounter,
    HrefDisplay, ElapsedTime, StyledText, SuggestLogin, and Themed for the
    stricter current React Native and TypeScript surface.

Dependency and tooling modernization:

  * Upgraded direct dependencies to current npm registry versions where the
    validation suite permits it, including TypeScript 6, current React
    Navigation packages, ESLint 10, Redux Toolkit 2.12, styled-components
    6.4, and the newest Expo 56 module releases.

    matrix after the Android release build proved the newer registry versions
    were not all compatible together. The app now uses React 19.2.3, React
    Native 0.85.3, Reanimated 4.3.1, Worklets 0.8.3, and Jest 29.7.

  * Held Babel on the newest 7.x release line. Babel 8.0.1 is current on npm,
    but the release Metro bundle failed because react-native-worklets still
    loads Babel 7-only preset code.

  * Removed @expo/webpack-config, @npmcli/fs, react-native-htmlview,
    @types/react-native-htmlview, @types/react-native, and
    react-native-settings-screen.

  * Overrode react-native-render-html's native-html render engine to the
    current @native-html/transient-render-engine 12.x line. That brings in the
    current CSS processor and removes the deprecated @types/react-native peer
    package from the reproducible npm graph.

  * Removed Hoot's direct fbjs dependency by replacing the hover helper's DOM
    capability check with a local typed globalThis probe.

  * Removed unused direct dependencies on react-is, redux-thunk, uglify-js, and
    patch-package. The empty patch-package postinstall hook was removed because

  * Added expo-background-task, expo-notifications, expo-system-ui,
    expo-task-manager, react-native-render-html, react-native-worklets,
    @react-native/jest-preset, @testing-library/react-native, ESLint, Jest

  * Updated the reproducible npm graph within the current Expo SDK 56
    compatibility range, including uuid 14.0.1 and
    @testing-library/react-native 14.0.1, current React Navigation 7.x patch
